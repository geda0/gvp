## Notes / rationale / flags

- **Ordering rationale.** The two NEW guards lead because they are the rigor the solo pass skipped and the highest red→green value (S1 = claim↔code traceability drift guard; S2 = the build-idempotency fence around the verified latent `open_resume`→`navigate_to_section` revert bug). The characterization slices (S4–S7) only LOCK copy that already ships, so they follow; S8 (qa) is last because a human verifies the very copy S4/S5 pinned.
- **Genuinely red vs characterization (load-bearing — tell the test-writer).** ONLY S2 is red because source is missing (the builder isn't exported yet); S3's named assertion currently passes against committed state (drift already fixed) and is demonstrated red by flipping the const. S1 and S4–S7 are characterization: the asserted content already exists, so they pass GREEN on write — the honest red is the BREAK-CHECK (delete/alter the asserted phrase or field and confirm the test fails), per the gate's red→green discipline. Do NOT let an implementer "drive" copy that's already there, and do NOT weaken a test if a claim reads thin — if a claim is FALSE against code, fix the COPY (none found false this cycle).
- **Highest regression-prevention value: S1.** Claim↔code traceability is the guard that prevents the marketing copy from drifting away from `.claude/hooks/tics-mcp.cjs` (naming a dropped tool, or claiming zero-dependency after a dep creeps in). S2 is a close second (the byte-stable build fence around the named latent bug). Both were entirely absent before this redo.
- **ADR-0012 non-negotiables.** S1: assert the FIXTURE (product contract), keep the kit read as a skip-if-absent tripwire on slow-moving facts only — do NOT couple the product suite to generated-kit file layout (it is refreshed by `ttics` upgrades). S2: prove `committed == builder output` with the builder's OWN exported code, in-process — NO shell-out (a unit test must not write into the tree), NO re-deriving the FAQ/build logic in the test (that only proves the test agrees with itself). If the export-refactor is deemed out of scope, ADR-0012's fallback is a read-only temp-`outputDir` sandbox import — still the real builder, still no in-tree write; default is the refactor (it also makes the builder generally unit-testable).
- **Don't duplicate the existing test.** `test/team-tactics-project.test.mjs` already pins: featured/leads/image (first test), the contact-CTA contract, chatSummary length + not-a-card-duplicate + projects.json sync, and the 0.56 MCP/cross-tool/tic_emit/opt-in/hook-only copy (incl. the negative that other-tool agents can't emit signal/block/commit). S4–S7 ADD only the gaps (zero-dependency/JSON-RPC/stdio/no-SDK phrasing in card+description; the non-empty presentation-field invariant). The implementer/test-writer must extend in place, not re-assert what's there.
- **Slices I'm unsure should exist (flagged, not padded).** S3 could be a sub-assertion inside S2 rather than a numbered slice — I kept it separate because ADR-0012 invariant #3 explicitly wants the `resume-access ⇒ navigate_to_section` regression named INDEPENDENTLY of the byte-equality check (so the red test names the bug); the navigator may fold it into S2 if a leaner queue is preferred. S4 overlaps the existing first team-tactics test (featured/image/leads) — it exists only to add the AC-1 "names the engineering" card-copy assertion; if that single assertion is added to S5/S7 instead, S4 can drop. S8 is the one qa-only slice (AC-2 "reads as credible engineering") and cannot be a unit test.
- **New invariants to record (per ADR-0012 §Invariants).** Three durable rules this redo introduces, for `docs/tdd/project-invariants.md`: (1) product suites depend on product fixtures, not generated-kit internals (kit reads are skip-if-absent tripwires on slow-moving facts); (2) idempotency is proven by the builder's own code against committed output — never by shelling out or re-deriving; (3) `resume-access ⇒ navigate_to_section` in both the `FAQ` const and committed `faq.json`, pinned by a named test.
